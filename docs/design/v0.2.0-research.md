# v0.2.0 — Enhancement Research

> **Status: research artifact, not a public commitment.** Items marked
> SHIP target v0.2.0; items marked DEFER target later versions; items
> marked DROP stay declined. Plans shift, priorities change. The only
> guarantee for end users is what's in the current PyPI release.

This document complements [ROADMAP.md](../ROADMAP.md) (high-level
milestones) and [CHANGELOG.md](../CHANGELOG.md) (what shipped). It surveys
every plausible v0.2.0 candidate, evaluates feasibility, and recommends a
concrete implementation set. The work was done in a single research pass
on 2026-04-29 with v0.1.1 freshly released.

## Methodology

Each candidate was evaluated by a combination of:
- **Code audit** — reading the relevant module(s) in `src/mps_sdpa/` and
  cross-referencing tests in `tests/`.
- **Documentation lookup** — current torch / Apple silicon / GitHub Actions
  references via Context7 and direct doc reads.
- **Agent-assisted research** — dispatched parallel research agents for
  CI/infra deadlines and the test-coverage audit, both kept generic and
  drop-in oriented.

For each item, the template captures:

| Field | Meaning |
|---|---|
| **Current state** | What exists in the code today; what already works; what fails |
| **Proposed change** | What we'd do if we shipped this |
| **Blockers / risks** | Technical unknowns, dependencies, gotchas |
| **Effort** | S (<1d), M (1–3d), L (3–7d), XL (1+wk) |
| **Value / impact** | L / M / H — how worthwhile this is for the maintainer's primary use |
| **Risk** | low / medium / high — likelihood of going sideways |
| **Recommendation** | SHIP v0.2.0 / DEFER vX.Y.Z / DROP / DOCUMENT |

## Candidates by area

### Capability / feature gaps

#### #1 — `torch.compile` compatibility

- **Current state.** The C++ extension is loaded via
  `torch.utils.cpp_extension.load()` and exposed through a custom
  `torch.autograd.Function` (`_ZCSDPAFunction`) whose `forward` and
  `backward` call into the pybind11 module directly. Under
  `torch.compile`, Dynamo emits a `UserWarning` for un-traceable third-
  party C/C++ extensions and either falls back via graph break or fails
  fullgraph compilation. The eager path is fully verified; the compile
  path is untested.
- **Proposed change.** Re-architect the public entry point as a
  `torch.library.custom_op("mps_sdpa::sdpa_forward", mutates_args=())`
  with a registered fake-tensor kernel for shape propagation, and use
  `torch.library.register_autograd` for the backward. The C++ extension
  surface stays unchanged — only the Python wrapper changes. This is
  the modern PyTorch-recommended pattern for making C++ extensions
  Dynamo-compatible.
- **Blockers / risks.** (a) The fake kernel must correctly propagate
  shape and dtype for all combinations of `attn_mask`, `is_causal`,
  `dropout_p`, and `scale`. (b) `register_autograd`'s `setup_context` /
  `backward` must work with the existing manual-softmax-bwd C++ kernel.
  (c) Compile path numerics must match the eager path within the
  documented tolerance — verify against the existing
  `cross_impl_atol` test infrastructure. (d) GQA fallback in `api.py`
  uses Python branching that Dynamo can trace but may produce graph
  breaks; consider whether to mark the GQA path as `@torch.compiler.disable`.
- **Effort.** M (1–3 days). Mostly pattern-following from the docs.
- **Value / impact.** M. The maintainer's primary workflows don't use
  `torch.compile`, but it's increasingly the default for PyTorch users.
  Closes a documented "What doesn't work" item.
- **Risk.** medium. Fake-kernel logic for masked / dropout paths needs
  careful matching to the runtime; numeric divergence is the failure
  mode to watch for.
- **Recommendation.** **SHIP v0.2.0.** Matches Option B framing — light
  touch to a current limitation. Existing tests gain new
  `@parametrize`'d `compile=True/False` runs to lock in the equivalence.

#### #2 — GQA acceleration (`Hq != Hkv`)

- **Current state.** `api.py` detects `Hq != Hkv` and routes to stock
  via `repeat_interleave` on K/V with a one-time `RuntimeWarning`.
  The fused `MPSGraph.scaledDotProductAttention` op is MHA-only on
  current macOS releases.
- **Proposed change.** Either (a) wait for an Apple-side GQA op (no
  control over timeline), or (b) build a hand-rolled fused GQA path —
  reuse the existing unfused-with-dropout backward pattern, but for
  GQA shapes, compute a per-group attention matrix without K/V
  expansion. The `repeat_interleave` we do today materializes the full
  Hq-shape K/V; a hand-rolled path could skip that allocation.
- **Blockers / risks.** Hand-rolled GQA needs its own forward AND
  backward graph in both the C++ extension and the pyobjc fallback —
  two graph types, two cache keys. Memory-saving is the only win since
  Apple's matmul is already optimal; we'd need to verify the saving is
  big enough to matter for typical GQA shapes.
- **Effort.** XL (1+ week). New graph types in two backends, new tests,
  benchmark to prove it.
- **Value / impact.** M. GQA is increasingly common in modern
  architectures, but the current `repeat_interleave` fallback is
  correct (just less memory-efficient).
- **Risk.** medium. New code paths in the Obj-C++ extension increase
  the surface area and the JIT compile time.
- **Recommendation.** **DEFER v0.3.0.** Already on the v0.3.0 list per
  ROADMAP. v0.2.0 ships polish; GQA acceleration is meaningful work
  that deserves its own focus.

#### #3 — Wider dropout fast-window

- **Current state.** The unfused-dropout MPSGraph wins over stock only
  when the attention-matrix bytes (`Lq * Lkv * elem_size`) fall in the
  16 MB – 64 MB range. Below: per-call overhead dominates. Above: stock
  wins (and the unfused build can OOM at very long sequences). Outside
  this window the dispatch falls back to stock.
- **Proposed change.** Profile to find the actual bottleneck at the
  upper bound. Hypotheses: (a) materializing the full `[Lq, Lkv]`
  attention matrix is the cost — that's algorithmic, can't fix without
  a tiled kernel. (b) The unfused graph's intermediate buffers retain
  memory across calls — could clear with `clear_graph_cache`. (c)
  The dropout mask itself is a large bf16 tensor; could be packed.
  Lower bound is less interesting; small shapes are fast on stock.
- **Blockers / risks.** Without profiling we can't predict the gain.
  This is genuinely a research item — the answer might be "no
  significant headroom available."
- **Effort.** M (1–3 days) for profiling + small optimizations; L if a
  tiled approach is needed.
- **Value / impact.** L–M. Dropout is standard during training but most
  modern architectures use small `dropout_p` (0.1) and the byte-range
  outside the window is a minority of training calls.
- **Risk.** low. The change would be an opt-in widening; current
  fallback path remains intact.
- **Recommendation.** **DEFER v0.3.0.** Already on the v0.3.0 list per
  ROADMAP. The "research first, decide later" framing fits — without
  profiling data this is speculative. Suggest doing a focused profiling
  pass as the v0.3.0 kickoff.

#### #4 — Second-order gradients (`create_graph=True`)

- **Current state.** Both backends use `@once_differentiable` on the
  backward, which raises a clear error rather than silently produce
  wrong values when a caller passes `create_graph=True`. The C++
  extension's `sdpa_backward` is opaque to autograd by design.
- **Proposed change.** To support double-backward, the backward would
  need to be expressed in terms of operations autograd can re-
  differentiate — either re-implementing in pure torch ops (losing the
  MPSGraph speedup on the bwd path entirely) or building a `d²L/dQ²`
  graph alongside the existing backward graph (complex).
- **Blockers / risks.** Trade-off is bad: pure-torch reimplementation
  would regress training perf significantly; manual second-order
  graphs are substantial new code.
- **Effort.** XL+ (multi-week). A real engineering investment.
- **Value / impact.** L. Second-order gradients are a niche need
  (meta-learning, certain regularizers). Most training workflows
  don't use them.
- **Risk.** medium. Numerics for second-order are subtle — high-effort
  change with limited audience.
- **Recommendation.** **DEFER conditional.** Stay in the "research /
  conditional" bucket of ROADMAP. Reconsider if a clear use case
  shows up.

#### #5 — fp32 acceleration on M3 Max

- **Current state.** On M3 Max, calibration finds stock fp32 SDPA wins
  at all probe shapes (256² / 1024² / 2048²). The calibrator returns a
  sentinel `2**40` (1 TiB) byte threshold, effectively disabling
  `mpsgraph_zc` for fp32 on M3 Max. Stock is already fast for fp32 on
  this chip — no defect, the calibrator is making the right choice.
- **Proposed change.** None functional. Two small polish items: (a)
  replace the `2**40` sentinel with `null` or a `disabled: true` flag
  in `thresholds.json` for clarity (this rolls into #9 calibration
  improvements). (b) Document the M3 Max fp32 behavior in COMPAT.md
  (already done in v0.1.1).
- **Blockers / risks.** None — already-correct behavior.
- **Effort.** S (small, rolled into #9).
- **Value / impact.** L. fp32 attention is rare in transformer
  training; bf16/fp16 are the usual paths.
- **Risk.** low.
- **Recommendation.** **DOCUMENT.** Already done in v0.1.1's COMPAT
  table. Cleanup of the sentinel value happens via #9.

#### #6 — Backward-graph optimization

- **Current state.** Backward graph: 4 matmuls + 1 reduce + several
  elementwise ops, plus a recompute of `Q@K^T → softmax` (live `attn_raw`).
  The recompute is intentional — caching `attn` from forward to
  backward would materialize the `[Lq, Lkv]` matrix and defeat the
  memory advantage. Apple's MPSGraph likely fuses some of the
  elementwise ops automatically.
- **Proposed change.** Possible micro-optimizations: (a) check if
  MPSGraph has a primitive for the softmax-bwd pattern (vs the manual
  `attn * (d_attn - sum(attn * d_attn))`). (b) Profile to confirm the
  matmuls dominate vs the elementwise ops. (c) Consider whether `dV`
  and `dK` can share intermediate computations (they share `attn^T`
  and `d_scores^T`).
- **Blockers / risks.** Without profiling, the wins are speculative;
  Apple's MPSGraph optimizer may already be doing these fusions.
  Manual softmax-bwd is the standard pattern across major attention
  implementations and is hard to beat without hardware-specific
  primitives.
- **Effort.** L (3–7 days) for profiling + experimentation.
- **Value / impact.** L. Training speedup is already 2–2.5×; further
  bwd optimization would yield small additional gains in a regime
  where stock is already 2.5× slower.
- **Risk.** medium. Refactoring the .mm extension's bwd graph is
  delicate; numeric drift between forward and backward could
  introduce regressions hard to spot.
- **Recommendation.** **DEFER conditional.** Only revisit if profiling
  during v0.3.0 dropout work suggests headroom.

### Robustness / UX

#### #7 — OOM recovery

- **Current state.** Both backends catch `RuntimeError` matching the
  string "out of memory" or "mps allocated", emit a `_log_fallback`
  reason, and route to a less memory-intensive backend (zc → pyobjc →
  stock). The string match is fragile to torch wording changes across
  versions. No test exercises the live OOM path.
- **Proposed change.** (a) Replace string match with a more robust
  detection (e.g., `torch.OutOfMemoryError` if torch ≥ 2.13, fallback
  to string match for older). (b) Add `torch.mps.empty_cache()` before
  the retry to free any cached blocks. (c) Add an integration test
  (parametrized by progressively larger `Lq`) that catches the OOM
  fallback on a memory-constrained device — though triggering OOM
  reliably on heterogeneous machines is fragile, so guard it behind
  an env var (`MPS_SDPA_RUN_OOM_TESTS=1`) and document. From the test-
  coverage audit, this gap was Gap 1 (highest priority).
- **Blockers / risks.** Triggering OOM in CI is unreliable; the test
  may be opt-in. The `torch.OutOfMemoryError` exception class is in
  recent torch but might not exist in 2.11 stable.
- **Effort.** S–M. Code change is small; making the test reliably
  trigger OOM is the harder part.
- **Value / impact.** M. OOM during training is genuinely common at
  long sequences; making the recovery path more robust pays off.
- **Risk.** low.
- **Recommendation.** **SHIP v0.2.0.** Pair with #17 test-coverage
  improvements.

#### #8 — Fallback diagnostics

- **Current state.** The unified `_log_fallback(reason)` emits messages
  in the format `[mps_sdpa.mpsgraph] falling back to stock: <reason>`,
  silent by default, opt-in via `MPS_SDPA_LOG_FALLBACKS={1,info,warn}`.
  `api.get_call_stats()` returns dispatch counters by backend name
  (`mpsgraph_zc`, `mpsgraph`, `stock_fallback`, `gqa_fallback`, etc).
  Reason strings are descriptive but not aggregated.
- **Proposed change.** Add per-reason fallback counters alongside the
  per-backend counters. Expose via `api.get_fallback_stats()` returning
  a dict like `{"short-seq": 134, "dropout outside window": 12,
  "OOM recovery: RuntimeError": 1}`. This lets users (and the
  maintainer) see at a glance why their hot path is going to stock.
- **Blockers / risks.** Reason strings need to be consistent (currently
  some include shape values like `"short-seq (2097152 < 4194304 bytes)"` —
  those should bucket to a single counter, not one-per-shape).
- **Effort.** S. Add a `_fallback_counters: Counter` and increment in
  `_log_fallback`; expose via api.
- **Value / impact.** L–M. Useful for performance debugging; the
  maintainer's CI already captures this via `print_call_stats`.
- **Risk.** low.
- **Recommendation.** **SHIP v0.2.0.** Cheap, modest improvement.

#### #9 — Calibration improvements

- **Current state.** Calibration runs 3 probe shapes (L=256, 1024, 2048)
  for each of bf16/fp16/fp32 on first import, picks the smallest L
  where mpsgraph wins by ≥ 5%. If no shape wins, returns `2**40`
  (sentinel for "always fall back"). Cached per `(chip, os, torch)`.
  Skip via `MPS_SDPA_SKIP_CALIBRATION=1`, force via
  `MPS_SDPA_FORCE_CALIBRATE=1`.
- **Proposed change.** (a) Replace the `2**40` sentinel with a clearer
  representation in the JSON (e.g., a `"disabled": true` field per
  dtype, or `null`). (b) Add finer probe-shape granularity around the
  expected crossover (e.g., 384, 512, 768) so the threshold reflects
  actual crossover rather than rounding to powers of 2. (c) Add a
  warm-up pass before timing — current code does 3 warmups but more
  may help on chips with thermal ramp. (d) Optionally print a one-line
  summary on calibration completion if `MPS_SDPA_LOG_CALIBRATION=1`.
- **Blockers / risks.** Adding probe shapes increases first-import
  time. Need to balance vs the current ~0.5–1 s budget. JSON schema
  change should bump `_CACHE_SCHEMA_VERSION` to invalidate old caches.
- **Effort.** M (1–3 days).
- **Value / impact.** M. Cleaner calibration UX; better thresholds on
  edge-case chips. Maintainer-relevant given the M3 Max sentinel
  observation.
- **Risk.** low. Existing tests cover the calibration path; schema
  bump invalidates on-disk caches cleanly.
- **Recommendation.** **SHIP v0.2.0.**

#### #10 — Cold-start time

- **Current state.** First import compiles the .mm extension via
  `torch.utils.cpp_extension.load()`, taking ~6 s. Subsequent imports
  hit the `~/.cache/torch_extensions/` cache and are instant. Full
  graph cache builds happen on first call per shape (~hundreds of ms
  per unique shape).
- **Proposed change.** The cleanest fix is **#13 pre-built wheels** —
  ship the compiled .dylib in the wheel and skip JIT entirely. Other
  options (lazy compile on first MPS call; parallel-compile) just
  shift when the user pays the cost. Direct improvements to compile
  time (smaller .mm, fewer headers) are marginal.
- **Blockers / risks.** Coupled to #13. Standalone cold-start work is
  low-leverage.
- **Effort.** Subsumed by #13.
- **Value / impact.** L–M. Annoying for first-time users; invisible
  after first run.
- **Risk.** low.
- **Recommendation.** **SUBSUME into #13.** Don't track separately.

#### #12 — CI matrix expansion

- **Current state.** `tests.yml` runs on `macos-latest` (currently
  Apple silicon M1, macOS 15) with Python 3.11 only. PyTorch is
  resolved via `pip install -e ".[dev]"` which picks the latest
  stable matching `torch>=2.11.0`.
- **Proposed change.** Add a small matrix:
  ```yaml
  strategy:
    fail-fast: false
    matrix:
      python-version: ["3.11", "3.13"]
  ```
  on push/PR (catches Python C-API changes between minors). Add a
  separate nightly cron job exercising `{3.10, 3.11, 3.12, 3.13}` ×
  `{torch stable, torch nightly}` to catch ABI drift in nightly
  torch. Avoid running torch-nightly on every PR (noisy).
- **Blockers / risks.** macOS minutes are billed at a 10× multiplier
  vs Linux. For a public repo this stays in the free tier; for a
  private repo it's billable. Hosted Apple silicon runners are M1 only
  (no M3/M4 available); cannot CI on the maintainer's actual chip.
- **Effort.** S (<1 day). YAML edits only.
- **Value / impact.** M. Catches Python C-API drift (3.13 free-
  threading), torch ABI breaks, and per-version `cpp_extension.load`
  surprises. Lower bound is essentially "what we have now."
- **Risk.** low.
- **Recommendation.** **SHIP v0.2.0.** Cheap, useful regression net.

#### #13 — Pre-built wheels

> **Spike outcome (2026-04-30): DEFER #13 to v0.3.0.** The stable-ABI
> route (single wheel per Python version) is not viable for v0.2.0. The
> blocker is upstream and independent of our code: `torch/extension.h`
> includes pybind11, whose `pytypes.h` and `buffer_info.h` rely on
> `Py_buffer`, `PyTypeObject` internals, `PyList_GET_ITEM`,
> `PyTracebackObject`, `PyFrame_GetBack`, `PyInstanceMethod_*`, and
> `PyListObject` — all CPython internals excluded under
> `Py_LIMITED_API=0x03100000`. Compilation never reaches our ATen
> surface (`getMTLBufferStorage`, `runMPSGraph`, `getCurrentMPSStream`),
> which would have been the actual concern. Pre-built wheels would
> therefore require the per-(torch × python) matrix route (~16
> wheels/release) — declined for v0.2.0 due to maintenance cost.
> **Reconsider for v0.3.0** pending either pybind11 stable-ABI support
> (active upstream effort) or migration to a non-pybind11 binding
> (e.g., `nanobind` with `stable_abi=True`, which would also require
> dropping `torch/extension.h`). v0.2.0 keeps the sdist + JIT compile
> path as before.

- **Current state.** PyPI ships an sdist; users compile the .mm at
  install time via `cpp_extension.load()`. First import takes ~6 s.
  Required system deps: Xcode CLT, ninja (in `dependencies`), pyobjc
  for the fallback path.
- **Proposed change.** Add `cibuildwheel` to build `cp310–cp313` ×
  `arm64` macOS wheels. Use torch's stable ABI (`Py_LIMITED_API` +
  `TORCH_TARGET_VERSION`) to ship one wheel per Python version that
  works across torch minor versions. Verify each ATen call we use is
  in the stable ABI subset (the `getMTLBufferStorage` symbol is the
  one to check; if it's not stable-ABI, fall back to a per-torch-
  minor wheel matrix). Keep the sdist as a fallback for users on
  unsupported torch versions.
- **Blockers / risks.** (a) Stable ABI subset coverage for our ATen
  usage is uncertain; verification needed. (b) Per-torch matrix
  blows up wheel count: torch{2.11..2.14} × py{3.10..3.13} × arm64 =
  16+ wheels per release. (c) `MACOSX_DEPLOYMENT_TARGET` becomes a
  contractual floor — set conservatively (14.0 to match the build
  runner) and document. (d) Need a release-time verification step
  that the published wheel actually imports + runs self-test on a
  fresh venv.
- **Effort.** L (3–5 days) if stable ABI works; XL (1–2 weeks) if
  forced into per-torch matrix.
- **Value / impact.** M–H. Removes the 6 s first-import delay and
  the Xcode CLT requirement (both annoying for new users). High
  visibility improvement.
- **Risk.** medium. New build pipeline = new failure modes;
  wheel-vs-sdist behavior divergence is a real test surface.
- **Recommendation.** **SHIP v0.2.0 if stable ABI works**; otherwise
  DEFER to v0.3.0. Spike: verify stable-ABI coverage for `getMTLBufferStorage`
  + ATen usage in <1 day. If green, proceed; if red, defer.

#### #17 — Test coverage gaps

- **Current state.** 235 tests across 39 files; coverage is
  comprehensive but the test-coverage audit found three concrete gaps:
